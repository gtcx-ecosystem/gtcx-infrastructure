# Data Protection Impact Assessment (DPIA) — GTCX Compliance Substrate

**Effective date:** 2026-05-22
**Scope:** compliance-gateway HTTP API · NATS JetStream `gtcx.audit.>` subjects · audit-flush sidecar · WORM S3 audit bucket · KMS keys · operational + audit PostgreSQL databases
**Frameworks:** GDPR Art. 35 (Data Protection Impact Assessment), CCPA / CPRA equivalents, FATF KYC retention requirements per jurisdiction (see `03-platform/tools/compliance-data/jurisdictions.json`)
**Review cycle:** on-change (any new data category, processor, or transfer mechanism triggers a re-DPIA)

## 1. Why a DPIA is required

GDPR Art. 35 mandates a DPIA when processing is "likely to result in a high risk to the rights and freedoms of natural persons." The substrate processes:

- **Personal identifiers** (DIDs, national IDs, biometric template hashes) under TradePass operations
- **Financial transaction context** under PvP settlement
- **Geographic location** under GeoTag origin verification
- **Profile-style metadata** under GCI compliance scoring

All four categories together create the conditions for a high-risk assessment per the EDPB guidelines on Art. 35. CCPA / CPRA treat the same processing as "sensitive personal information" triggering an equivalent assessment.

## 2. Data flow map

```
Subject submits query  →  compliance-gateway  →  protocol tools (read-only)
                              │
                              ├── signed audit record  →  JetStream  →  audit-flush  →  WORM S3
                              │     (no PII in record body — payload is hashed,
                              │      tenantId is in the subject not the record)
                              │
                              ├── operational write  →  operational PostgreSQL
                              │     (current state; mutable; row-level encryption
                              │      for fields tagged sensitive)
                              │
                              └── audit write  →  audit PostgreSQL
                                    (append-only; separate IAM user;
                                     no UPDATE/DELETE grants)

                                  WORM bucket: Object Lock COMPLIANCE,
                                  2557-day retention, no possible deletion
                                  before retention expires regardless of
                                  IAM grants
```

## 3. Personal data categories

| Category                  | Example fields                                  | Purpose                                                 | Retention floor                                       | Lawful basis (GDPR Art. 6)                                            |
| ------------------------- | ----------------------------------------------- | ------------------------------------------------------- | ----------------------------------------------------- | --------------------------------------------------------------------- |
| Identity                  | DID, national ID number, name                   | TradePass operations: identity issue / resolve / verify | 5 years (KYC) per jurisdiction; up to 7 years (audit) | (b) contract performance — necessary to deliver the regulated service |
| Biometric template hashes | SHA-256 of fingerprint / face template          | Authentication of identity for high-value trades        | 5 years                                               | (b) contract; (f) legitimate interest (fraud prevention)              |
| Financial / transactional | Settlement IDs, parties, amounts, currencies    | PvP settlement, GCI scoring                             | 7 years (audit + financial reporting)                 | (b) contract; (c) legal obligation (FATF)                             |
| Geographic                | Lat/lon for commodity origin                    | GeoTag origin verification                              | 7 years                                               | (b) contract; (c) legal obligation                                    |
| Profile metadata          | GCI tier, compliance score, attestation history | GCI scoring + PANX attestation                          | 7 years                                               | (b) contract; (f) legitimate interest                                 |

Original payloads are NEVER stored in the audit chain (only a SHA-256 hash). The operational + audit databases hold the original data; the WORM bucket holds the hash-linked chain (no raw payload).

LLM providers receive the user-submitted query text. Per the `03-platform/tools/compliance-gateway/src/schemas.mjs` validator, query payloads are bounded (≤4096 chars query, ≤16KB context envelope) and wrapped in `---BEGIN UNTRUSTED CONTEXT---` delimiters that the system prompt instructs the model to treat as data.

## 5. Cross-border transfer mechanisms

The substrate's primary region is `af-south-1`. African jurisdictions vary on cross-border data transfer:

- **Zimbabwe:** Cyber and Data Protection Act (2021) — adequate safeguards required; POTRAZ notification
- **South Africa:** POPIA — adequate-protection assessment per §72
- **Nigeria:** NDPA / NDPR — DPO sign-off and notification for transfers
- **Kenya:** Data Protection Act 2019 — written consent or adequate-safeguards mechanism

For LLM providers in the US:

- **Mechanism:** SCCs (Standard Contractual Clauses, EU 2021/914 / UK IDTA equivalents) in vendor agreements
- **Supplementary measures:** delimited untrusted-context block prevents the substrate from leaking system-prompt information; per-principal token budget bounds the volume; cost telemetry captures every byte sent to a third-party processor for audit

**Open gap:** SCC mechanism execution must be confirmed by Legal Lead before pilot expansion. Annotated as P1 in the risk register below.

## 6. Risk register

| #       | Risk                                                                     | Likelihood | Impact | Mitigation                                                                                                                                                                                                                                                                       | Status        |
| ------- | ------------------------------------------------------------------------ | ---------- | ------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------- |
| DPIA-01 | LLM provider retains query text beyond contractual window                | Medium     | High   | Provider TOS audit; data-processing addenda in place; substrate logs per-request what was sent                                                                                                                                                                                   | Active        |
| DPIA-02 | Cross-border transfer mechanism (SCC) not executed for all jurisdictions | Medium     | High   | Legal Lead to confirm SCC execution per jurisdiction before pilot expansion                                                                                                                                                                                                      | Open (P1)     |
| DPIA-03 | Personal data reconstructed from `payloadHash` collision attack          | Very Low   | Medium | SHA-256 with 2^128 collision resistance; payload is a separate, encrypted-at-rest store; collision attack would not reveal plaintext                                                                                                                                             | Mitigated     |
| DPIA-04 | Audit DB read access bypasses operational permission model               | Low        | High   | Audit DB has separate IAM user, separate IRSA role; cross-tenant subject routing on JetStream (ADR-015); per-tenant prefix in WORM                                                                                                                                               | Mitigated     |
| DPIA-05 | Personal data in URL query strings ends up in tamper-evident audit chain | Low        | Medium | `03-platform/tools/compliance-gateway/src/audit-target.mjs` strips query strings + URL fragments + caps at 200 chars before signing                                                                                                                                              | Mitigated     |
| DPIA-06 | Audit retention exceeds jurisdiction's data minimization requirement     | Low        | Medium | Per-jurisdiction retention floors in `03-platform/tools/compliance-data/jurisdictions.json`; WORM bucket Object Lock at 2557 days satisfies the longest retention floor (7yr South Africa Companies Act); shorter-retention jurisdictions accept the longer floor as permissible | Mitigated     |
| DPIA-07 | Subject access request (GDPR Art. 15) can't surface all data             | Low        | Medium | Operational DB has per-DID indexing; audit DB has per-DID indexing; WORM chain is verifiable but lookup-only via JetStream subject (not text-searchable); compensating control: log-shipping pipeline mirrors WORM content to a searchable index                                 | Open (P2)     |
| DPIA-08 | Erasure request (GDPR Art. 17) conflicts with FATF retention             | Very Low   | Medium | Operational DB supports erasure; audit DB + WORM bucket are append-only by design — erasure violates FATF Art. 11 retention; "right to be forgotten" yields to legal obligation per Art. 17(3)(b) and (e); subject notified of the legal basis for retention                     | Active policy |

## 8. Data subject rights and how the substrate supports each

| Right (GDPR / CCPA)                  | Substrate support                 | Operational path                                                                                                       |
| ------------------------------------ | --------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| Access (Art. 15)                     | Partial — see DPIA-07             | Operator query on per-DID index in operational + audit DB; runbook needed for WORM mirror                              |
| Rectification (Art. 16)              | Yes — operational DB              | Standard update flow; superseded record is preserved in audit chain                                                    |
| Erasure (Art. 17)                    | Conflicts with FATF — see DPIA-08 | Operational DB erasure executable; audit retention takes precedence per Art. 17(3)                                     |
| Portability (Art. 20)                | Yes                               | DID + claims exportable as a verifiable presentation via TradePass                                                     |
| Objection (Art. 21)                  | Yes                               | Provider override via `PREFERRED_PROVIDER` blocks LLM transfers if requested                                           |
| Automated decision opt-out (Art. 22) | Mitigated by SIGNAL S1            | Every mutating decision requires a human approval ticket; the substrate doesn't permit AI-only consequential decisions |

## 9. Conclusion + sign-off (pending)

The substrate is GDPR/CCPA defensible with the following conditions:

1. SCC execution for cross-border LLM transfers (DPIA-02, P1) — Legal Lead action item
2. WORM mirror to a searchable index for subject access requests (DPIA-07, P2) — Platform Engineering backlog
3. Periodic re-DPIA on any new processor, new jurisdiction plugin, or new data category
